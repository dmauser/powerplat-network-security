## Publish approach used

**ARM zip deploy** (`az functionapp deploy --src-path <zip> --type zip`). This uses `PUT .../sites/{name}/extensions/ZipDeploy` on the ARM management plane, bypassing the SCM/Kudu endpoint. Safe with `publicNetworkAccess=Disabled`.

**Fallback (if ARM zip deploy fails):** Run-from-Package — upload zip to `deployments` container in func storage account, generate user-delegation SAS, set `WEBSITE_RUN_FROM_PACKAGE` app setting to SAS URL. See `scripts/04-deploy-functions.ps1`.

## Learnings for history.md

- **Publish approach:** ARM zip deploy (`az functionapp deploy --type zip`) is the correct method for private SCM function apps — uses ARM management plane, not Kudu.
- **CallerIPAddress ranges:** Expected 10.10.2.X (east snet-funcapp) and 10.20.2.X (west snet-funcapp). Not live-verified due to quota constraint.
- **DNS/PE:** No gotchas — Trinity's Bicep already had all zones linked to both VNets. West-to-east KV path works by DNS zone link + VNet peering.
- **MCAP quota constraint:** MCAP internal subscriptions may have `Total VMs: 0` for App Service Plans. Affects all SKUs including EP1, S1, B1. Use Pay-As-You-Go subscription or request quota increase.
- **Raw REST vs Az.KeyVault:** Raw `Invoke-RestMethod` to KV REST API is recommended over Az.KeyVault module in PowerShell Functions because: (1) faster cold start (no module download), (2) auto-tracked as App Insights dependency (the SDK captures `Invoke-RestMethod` outbound calls explicitly).
